# 项目宪法 (Constitution)

## 核心原则

### 1. 代码质量
- **模块化优先**：每个模块单一职责，函数不超过 50 行
- **类型安全**：所有函数必须有类型注解，使用 mypy 严格模式
- **错误处理**：所有外部调用必须有异常处理和重试机制
- **日志记录**：关键操作必须记录日志，使用结构化日志格式

### 2. 测试标准
- **测试覆盖率**：核心模块覆盖率 ≥ 80%
- **单元测试**：每个公共函数至少一个单元测试
- **集成测试**：关键数据流必须有端到端测试
- **测试数据**：使用真实数据的子集作为测试 fixture

### 3. 用户体验一致性
- **响应式设计**：网页必须适配桌面和移动端
- **加载状态**：所有异步操作必须显示加载指示器
- **错误提示**：用户友好的错误消息，附带解决建议
- **性能要求**：页面加载 < 3 秒，图表渲染 < 5 秒

### 4. 性能要求
- **数据处理**：使用 Polars 进行大数据处理，避免 pandas 内存溢出
- **查询优化**：Parquet 文件必须使用分区和索引
- **缓存策略**：计算结果必须缓存，缓存有效期 24 小时
- **增量更新**：支持增量数据处理，避免全量重计算

### 5. 环境管理
- **Python 版本**：固定使用 Python 3.11+
- **虚拟环境**：所有开发必须在 venv 中进行
- **依赖锁定**：使用 requirements.txt 锁定所有依赖版本
- **环境隔离**：Claude Code 只能在虚拟环境下执行代码

### 6. 安全规范
- **数据安全**：不提交敏感配置（数据库密码、API 密钥）
- **输入验证**：所有用户输入必须验证和清理
- **路径安全**：文件操作必须限制在项目目录内
- **命令注入**：禁止执行未经验证的外部命令

## 开发治理

### 技术决策流程
1. **简单变更**（单文件修改）：直接实现
2. **中等变更**（多文件修改）：创建 spec 文档，评审后实现
3. **重大变更**（架构调整）：创建 RFC 文档，团队评审后实现

### 代码审查清单
- [ ] 代码符合模块化原则
- [ ] 有适当的错误处理
- [ ] 有单元测试覆盖
- [ ] 日志记录完整
- [ ] 性能影响已评估
- [ ] 文档已更新

### 版本控制
- **分支策略**：feature 分支开发，main 分支稳定
- **提交规范**：使用 Conventional Commits
- **发布流程**：语义化版本控制 (SemVer)

## 项目目标

构建一个基于网页的交易链路延时数据分析框架，具备：
1. **交互式数据探索**：支持多维度的数据筛选和可视化
2. **智能异常检测**：自动识别高延时聚集区和异常链路
3. **链路归因分析**：分析延时来源（TGW/柜台/网络）
4. **报告生成**：自动生成分析报告和图表
5. **可扩展架构**：支持新的数据源和分析算法
